## What's good

### Le contrat Pydantic est solide
`src/events/schemas.py` est l'achievement central. Les 5 models avec
`extra="forbid"` rendent impossible le drift silencieux producer ↔
consumer qui avait causé les bugs précédents. Trois trucs particulièrement
bien faits :

* **`mode='before'` validators** pour la tolérance casing : `BUY` accepté
  ET `buy` accepté, permet la migration progressive sans casser les
  producers legacy.
* **`@field_serializer` qui re-stringifie price/size** pour conserver le
  wire contract Decimal-as-string que les consumers existants attendent
  déjà.
* **Constantes `CHANNEL_*`** comme source unique : `CHANNEL_SCHEMA` est
  un mapping dict qui drive le bridge automatiquement.

### `WSBridge._assert_channel_coverage()` est une excellente guard
Vérifier au startup que `CHANNEL_SCHEMA` (producers) et
`CHANNEL_TO_WS_TYPE` (bridge) ont les mêmes clés évite la classe entière
de bug "producer publie sur un nouveau canal, bridge le reçoit mais le
forward comme `type: unknown`". Le crash explicite au démarrage est très
préférable à un silence en runtime.

### Rate-limit par canal via token bucket monotone
Implémentation simple, monotonic-clock (immune aux NTP), drop-counts
loggés périodiquement (pas de log spam). Bien pensé.

### `pillars_status` parallèle avec graceful fallback
Pattern propre : si on a un pool, on parallélise via `gather` avec
`return_exceptions=True`. Si on a une connection seule (legacy/tests), on
reste séquentiel. La détection pool vs connection via
`_looks_like_pool()` est duck-typing assumée. Le fallback
`_extract_pool_from_conn` est astucieux (utilise l'API privée
`_holder._pool` d'asyncpg, stable depuis 0.23) — mais c'est marqué
comme tel dans le commentaire, donc OK.

### LiveStore slice-based avec back-compat
`store.subscribe(fn)` legacy garde le full snapshot, `subscribeSlice` est
le nouveau fast-path. Les composants non-migrés continuent de
fonctionner ; les nouveaux components ne paient pas le re-render global
sur un trade qui ne les concerne pas. La transition est progressive sans
forcing.

### Tab keep-alive avec lazy-mount via ref
Le pattern `visitedTabsRef = new Set([tab])` computé pendant render
plutôt que dans un effect évite le flash of empty content au premier
switch. C'est subtil mais correct.

### Cache thrash detection
`_cached_helper` logue automatiquement `cache thrash detected` quand le
rebuild prend plus de la moitié de la TTL — surfaces les helpers à
re-tuner sans avoir à les profiler en isolation. Pattern à généraliser.

#### 2. `maintenance_loop` rebuild 130-161s
Source réelle du "dashboard figé" intermittent (cf
`docs/perf/ws_lag_diagnosis.md` §5.2). Le snapshot live est rebuild toutes
les 30s mais le rebuild prend parfois >TTL, ce qui veut dire qu'il y a
des fenêtres où la donnée affichée a 3+ minutes.

**Action recommandée** : profiler le rebuild (probable cause : un des
helpers cachés a un cold-start qui touche `book_quality_snapshots` sans
index). Diviser le snapshot en sous-builds parallèles avec leur propre
cache (déjà partiellement fait dans `_HELPER_CACHE_TTLS` pour
inspector/market_scanner — généraliser).

#### 3. `book_quality_snapshots` partitioning
4.3M rows/day, 390M à 90 jours, sans partitioning. La table devient hot
pour toute query qui agrège dessus (et il y en a — `_check_oracle` dans
`pillars_queries.py` la touche par exemple).

**Action recommandée** : migration SQL pour `PARTITION BY RANGE (closed_at)`
en partitions journalières ou hebdomadaires + job de drop des partitions
>90 jours.

## What's worth a vigilance

### A. La règle "un fait / un service" est facile à casser
Les agents qui rajoutent un nouvel endpoint copient souvent une query
existante "for context". Le grep en revue (Règle 1 du
`CODE_REVIEW_RULES.md`) est non-négociable, sinon dans 6 mois on aura à
nouveau 3 endpoints qui répondent à des compteurs subtilement
différents.

### B. Le polling adaptatif peut masquer une régression du WS
Si le WS commence à drop tous les events typés silencieusement (par
exemple parce qu'un nouveau canal a été ajouté à `CHANNEL_SCHEMA` mais
pas à `CHANNEL_TO_WS_TYPE` → fail au startup → bridge ne tourne pas), le
front se met à poller à 10s et la dashboard a l'air "presque" normale.

**Mitigation** : la sidebar montre `WS: OFFLINE` dans ce cas (cf
`connectionState`), donc visible. Mais le warning log
`WSBridge: typed handler called on unmapped channel` doit être surveillé
en prod (Telegram alert quand il apparaît plus de N fois en 1h).

### C. Le casing tolerance peut devenir une dette
Les `_uppercase` validators acceptent `buy`/`BUY`, `running`/`RUNNING`.
Tant qu'on a des producers legacy qui émettent lowercase, c'est utile.
Mais au moment où tous les producers sont migrés, ces validators
deviennent du code mort qui masque le drift. Re-vérifier dans 6 mois.

### D. `_assert_channel_coverage` ne vérifie que les noms, pas les schémas
Si un producer publie un payload qui valide mais avec un champ
`extra="forbid"` n'autorisant pas un nouveau champ, le bridge logue un
`dropped malformed event` et le front ne reçoit rien. La guard de
startup ne catch pas ça.

**Mitigation** : un test d'intégration round-trip producer ↔ consumer
pour chaque canal devrait être ajouté (déjà présent pour
`trades:observed`, à généraliser).

### E. `decisions.action` accepte canonique ET legacy
Le Literal autorise `OPEN, CLOSE, REDUCE, SKIP, follow, fade, skip, volume_anticipation`. C'est 8 valeurs qui sont en réalité 2 vocabulaires
parallèles. Risque : un nouveau dev ajoute `BUY` (raisonnable mais hors
des deux vocabulaires) et ça raise.

**Mitigation** : c'est intentionnel et documenté dans le docstring du
model. Mais à terme, choisir un vocabulaire et migrer.

### F. La debounce `system:status:last_emit` est par-process
`queries.system_status()` debounce les events `SystemStatusChanged` via
un GET/SET Redis. Si plusieurs workers FastAPI tournent (gunicorn -w 2+),
chacun peut publish son signal séparément, le debounce est partagé donc
le second wins. Pas un bug, juste un comportement à connaître.

### G. La maturity score peut piéger l'opérateur
`overall_pct = mean(4 signals)` peut être >70% (banner se cache) alors
qu'un des 4 signaux est à 0%. Cas typique : `decision_health_pct=0.0` (24h
sans décision actionnable, situation valide en bootstrap) mais les 3
autres signaux sont à 100% → `overall_pct = 75%` → banner cachée.

**Mitigation** : peut-être passer en `min()` au lieu de `mean()`, ou
ajouter un seuil "any below 10% keeps banner visible". À discuter avec
l'opérateur.

## Conclusion

Le refactor a livré ce qu'il avait promis : les valeurs affichées à
l'écran sont maintenant calculées une fois et propagées partout, les
producers Redis sont typés sur les 5 canaux les plus chauds, le
dashboard ne polle plus la live data qui a un producer. La dette
résiduelle est claire et chiffrée (7 producers à migrer, 2 god modules à
splitter, 1 table à partitionner).

Le point critique à surveiller en production : le warning
`WSBridge: typed handler called on unmapped channel` doit être à zéro.
S'il apparaît, c'est qu'un agent a contourné `_assert_channel_coverage`
ou a ajouté un channel sans mettre à jour le bridge.

Les trois règles du `CODE_REVIEW_RULES.md` capturent l'essence du
refactor. Si elles sont appliquées en revue de chaque PR, la dette ne
revient pas.
